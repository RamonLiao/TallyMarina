## 1. 背景與問題定義

### 1.1 市場背景

隨著穩定幣與鏈上資產在企業端的使用增加，傳統 ERP / 會計系統無法直接理解錢包、交易所、DeFi 協議與多鏈資產的原始資料，財務團隊往往靠 Excel 與人工對帳維持財報正確。[][]

因此出現一整類「crypto subledger / digital asset accounting」產品，負責在「鏈上資料」與「ERP / 総帳系統」中間，提供：

- 錢包／交易所／託管資料整合與標準化
- 交易分類、成本基礎與公允價值計算
- 依 IFRS / GAAP 政策產生分錄
- export 到 NetSuite / SAP / Xero 等系統使用的 journal entries[][][]


### 1.2 痛點

1. **Crypto 活動難以關帳**：多鏈、多錢包、多交易所，加上 DeFi 合約，月關帳常常要花 2–3 週整理與對帳。[][]
2. **審計與合規風險**：缺乏可重現的 subledger 與規則引擎，會計判斷散落在 Excel 與 side-channel 討論。[][]
3. **AI 工具與會計流程缺乏結合**：AI 能讀交易、分類，但缺乏可審計、可控的決策邊界與會計政策落地。
4. **Sui / 新鏈 coverage 不足**：現有產品對 Sui 這種物件導向、高 TPS L1 支援有限，專門的 data / accounting 引擎仍是空白。[][]

***

## 2. 產品定位與目標

### 2.1 定位（一句話）

> **Sui Agentic Subledger** 是一個專為財務團隊打造的數位資產子分類帳，利用 Sui Stack 與 AI，將錢包與 DeFi 活動轉換為 IFRS / GAAP-ready 的分錄與審計軌跡，並串接既有 ERP。

### 2.2 主要客群（ICP 摘要）

- L1：正式營運中的 Sui 項目方、crypto-active company（CEX、做市商、基金等）
- L2：使用穩定幣收付款、treasury 管理的 Web2 企業（跨境 B2B、遊戲、SaaS 等）


### 2.3 產品目標（MVP）

1. 把 **Sui 錢包 / 部分 CEX** 的交易資料，轉成可審查的 accounting events。
2. 讓財務人員在 Web 介面中，利用 AI 建議 + 規則引擎，產生可 export 至 ERP 的 journal entries。
3. 每期關帳時，能產出 **reconciliation** 報表與 **Walrus 上的 audit snapshot**。

關鍵指標（examples）：

- 月關帳時間縮短至少 50%。
- 90% 以上交易可自動歸類且僅需輕量 review。
- 對帳差異控制在總規模的 1% 以內。

## 3. 使用者與角色

### 3.1 主要角色

- **財務經理 / CFO**
    - 需求：掌握數位資產整體風險、關帳時間、審計準備、報表的可信度。
- **會計人員 / 外包會計師**
    - 需求：把雜亂的鏈上活動變成可檢視、可調整、可 export 的分錄與報表。[]
- **Treasury / Ops / Protocol 團隊**
    - 需求：看錢包與 DeFi 部位狀況，確保跟 subledger / ERP 維持一致。
- **審計人員**
    - 需求：查閱 period-end positions、reconciliation、Walrus audit packs 與規則版本。

***

## 4. 核心使用情境（User Stories）

### 4.1 連接 Sui 錢包並載入交易

- 身為財務人員，我可以輸入 Sui address / 使用 dApp Kit 連接企業錢包（由內部 ops 簽名授權），系統會自動拉取過去 N 個月交易與餘額。
- 系統會定期從 Sui GraphQL / Indexer 更新最新交易。


### 4.2 清洗與事件分類

- 系統會依 built-in parser + AI 建議，把各種交易標記為收款、付款、swap、staking、fee、內部轉帳等，並顯示信心分數。
- 不確定或低信心的交易會放入「需 review」Queue，避免自動入帳。


### 4.3 應用會計政策與規則

- 財務經理可以選擇 IFRS / GAAP 模板，並設定公司政策（stablecoin 分類、成本基礎方法等）。[][]
- 系統根據政策，產生 draft journal entries 供人員審查（不需自己寫分錄）。[][]


### 4.4 Review 與 Close

- 會計人員在 UI 中檢視每筆事件 + 對應分錄，使用「接受、修改、退回」操作。
- 所有 review 行為都有 log，以利審計。
- 完成後可以一鍵生成：
    - 當期 subledger journal export（ERP 格式）
    - reconciliation 報表
    - Walrus 上的 audit snapshot link


### 4.5 審計查核

- 審計人員可以依期間、實體查看：
    - positions / balances
    - reconciliation 差異與說明
    - 對應的 Sui transaction hashes
    - 對應 Walrus snapshot（確保數據未被事後竄改）

***

## 5. 範圍與不在範圍（MVP）

### 5.1 MVP 範圍（In Scope）

- 鏈上來源：
    - Sui L1（錢包 + 部分協議，例如 DeepBook 基本 swap / trade）。
- 其他來源：
    - 手動 CSV 匯入一兩個 CEX（例如 Binance, OKX 的 generic 對接）。
- 功能：
    - 資料接入與標準化（RawTransaction → NormalizedEvent）。
    - AI-assisted 事件分類（只建議，不直接入帳）。
    - 會計政策設定（高階；詳細規則另見會計規格書）。
    - 規則引擎產出 journal entries（限少數事件類型）。
    - Reconciliation（wallet ↔ subledger）。
    - CSV / ERP-ready journal export。
    - Walrus-based audit snapshot（基礎版）。


### 5.2 不在範圍（MVP 之外，後續版本）

- 完整多鏈 / 全交易所支援。
- 全面報表引擎（完整三大表、儀表板）。
- 原生 NetSuite / SAP API Connector。
- 全自動 tax / global regulatory reporting。
- 多實體 transfer pricing / consolidation。

## 6. 功能模組（非會計邏輯版）

> 補充：會計邏輯（資產分類、估值、分錄規則）由「會計規格書」詳細定義，這裡只描述系統看得到的 interface 與行為。

### 6.1 Data Ingestion 模組

**目的**：與 Sui、CEX 等資料來源連線，取得原始交易與餘額。

**需求：**

- 透過 Sui TypeScript SDK + GraphQL / Indexer 抓取：
    - 指定 address / object 的交易列表、餘額與 meta。[][][]
- 支援配置：
    - 同一 entity 可有多個 source（錢包 / exchange account / custody）。
- 定期更新：
    - 排程 job（例如每 10 分鐘）抓最新交易。
- 錯誤處理：
    - 網路錯誤 / 節點超時時重試，並記錄錯誤 log。

**輸出**：RawTransaction 資料表。

***

### 6.2 Normalization \& Classification 模組

**目的**：將 RawTransaction 轉為統一的 NormalizedEvent，並附上 AI 建議與信心分數。

**需求：**

- 支援以下 event_type：`ASSET_RECEIPT`, `ASSET_DISPOSAL`, `INTERNAL_TRANSFER`, `SWAP`, `STAKING_DEPOSIT`, `STAKING_REWARD`, `FEE_GAS`, `UNCLASSIFIED_CONTRACT_INTERACTION` 等。
- 每筆事件包含：
    - 基本欄位（asset, quantity, direction, tx_hash, block_time, wallet_address）。
    - `economic_purpose`（如 customer_receipt, vendor_payment, treasury_reallocation）。
    - `confidence_score`（0–1）。
    - `review_status`（pending / approved / rejected）。
- AI 模組：
    - 對每筆事件提供建議的 `event_type` 與 `economic_purpose`。
    - 低信心事件自動進入「需審查」佇列。

***

### 6.3 Accounting Policy \& Rules Engine（接口版）

**目的**：依 entity 的會計政策與預設規則，將 NormalizedEvent 轉為 journal entries。

**此處只定義外部介面：**

- Input：`NormalizedEvent`, `PricePoint`, `PolicySet`, `PositionLots`。
- Output：`JournalEntryHeader + Lines`（含 debit, credit, account_code, amount, currency）。
- 行為：
    - 僅對 `review_status = APPROVED` 的事件運行。
    - 若規則找不到對應 mapping，標記為「需要政策配置」，不自動產生 JE。

> ⚠ 詳細：
> - IFRS / GAAP 具體處理（如 IAS 38, IFRS 9, ASU 2023‑08）
> - 成本基礎方法（FIFO / WAC 等）
> - 各 event_type 對應科目的 mapping
>
> 統一挪到「會計規格書」詳細說明。

***

### 6.4 Review \& Approval Workflow 模組

**目的**：提供財務人員一個可以查看、調整與核准交易與分錄的介面與流程。[]

**需求：**

- Transaction Review Queue：
    - 顯示事件列表（含 AI 建議、信心分數、對應分錄 preview）。
- Event 詳細頁：
    - 顯示原始鏈上 tx（含 Sui Explorer link）、NormalizedEvent、AI 建議。
    - 允許修改 `event_type`, `economic_purpose`, memo 等。
    - 操作：Approve / Reject / Flag。
- Journal Review：
    - 針對某事件看對應 JE lines，允許調整 account_code、memo、dimension。
- Audit Log：
    - 所有變更紀錄：誰在何時對哪個欄位做了什麼操作。

***

### 6.5 Reconciliation 模組

**目的**：比較 on-chain balances / exchange balances 與 subledger / ERP balances，產生差異報告。[][]

**需求：**

- 支援按以下維度對帳：
    - entity + wallet + asset + period。
- 計算：
    - source_balance（Sui / CEX）。
    - subledger_balance（sum of JE for that scope）。
    - variance_qty / variance_value。
- 顯示：
    - 差異列表，允許填寫 exception reason / resolution。
- 支援 export Reconciliation Report（CSV / PDF）。

***

### 6.6 ERP / GL Integration 模組（MVP）

**目的**：將已核准的 journal entries 匯出到現有 ERP / accounting system。

**MVP 需求：**

- ERP export format：
    - 先支援 generic CSV（可 mapping 到 NetSuite / Xero / QuickBooks 的 journal import 欄位）。
- Export 行為：
    - 允許分期 export（例如只 export 某 period / 某 entity）。
    - export 後，為該 JE 標記「exported」。
- 後續版本：
    - 建立 specific ERP connector（API-level integration），另於未來規格書擴充。

***

### 6.7 Audit Trail \& Walrus Snapshot 模組

**目的**：將每期關帳的 subledger state 與 reconciliation 結果存儲在 Walrus，形成不可竄改的審計檔案。

**需求：**

- Period Close 操作時：
    - 將指定 entity + period 的：
        - positions / balances
        - Journal entries
        - Reconciliation 差異與註解
    - 打包成一個壓縮檔或結構化檔案（JSON / parquet）。
- 寫入 Walrus：
    - 使用 Walrus API 上傳，取得 content ID / reference。
- 回寫本系統：
    - 在 `AuditSnapshot` 表中儲存 Walrus CID，並連結到 period、entity。
- UI：
    - 在 UI 中可看到「View Walrus Snapshot」按鈕，點擊可檢視／下載檔案。

***

### 6.8 AI Assistant 模組（限制版）

**目的**：提供財務人員可用的 AI 協助，但不直接自動 posting。

**MVP 能力：**

- 建議事件類型與經濟目的。
- 對可疑交易提出「此交易可能是內部轉帳 / DeFi interaction / spam」提示。[][]
- 協助生成 memo / 描述（例如會計備註）。

**限制：**

- AI output 只能作為 suggestion，不可在未經人工核准情況下直接進入規則引擎。

***

### 6.9 Security \& Permissions

**核心需求：**

- 支援 Role-based Access Control (RBAC)：
    - Admin（設定政策、權限、來源）
    - Accountant（review / edit / approve JE）
    - Viewer（只讀）
- 所有變更需有完整 Audit Log。
- 重要憑證（如 CEX API Key）需加密儲存。

***

## 7. Sui Stack 整合設計（摘要）

**會用到的 Sui Stack 元件：**

- **TypeScript SDK / Clients**：連接 Sui RPC / gRPC，抓取錢包與交易資料。[][]
- **GraphQL RPC \& Indexer**：提供結構化、可篩選的歷史資料查詢（支撐 reconciliation / 報表）。[][]
- **DeepBook（選配）**：作為企業交易活動的一個來源（解析成交、手續費等）。
- **Walrus**：儲存 period-end audit snapshots，確保關帳資料不可竄改。[]
- **SuiNS / Seal / zkLogin（視時間）**：改善 UX（可讀地址、加密訊息與 Web2 友善登入）。[page:1][]

***

## 8. 非功能性需求（初版）

- **性能**：
    - 能處理至少每月 1–5 萬筆 Sui 交易的 ingestion 與分類。
- **可用性**：
    - Web UI 主要針對桌機使用（財務人員）；手機僅要求可閱讀。
- **可靠性**：
    - 所有 accounting output 必須可重跑（同樣輸入 ＋ 同樣政策 → 同樣 output）。
- **安全**：
    - 支援 2FA / SSO（後續版本）。
    - 加密保存敏感資料（API keys、密鑰等）。

***

## 9. 路線圖（高階）

### Phase 0（Hackathon MVP）

- 單一 entity + Sui 地址 ingestion。
- 基本事件分類（收款、付款、swap、fee）。
- 簡易 AI 建議與 review UI。
- 規則引擎支援少數事件類型。
- CSV export + Walrus snapshot（minimal）。


### Phase 1

- 增加 staking / CEX 交易 ingest。
- 強化 reconciliation 模組。
- 增加會計政策 UI 與 IFRS / GAAP 模板。


### Phase 2

- 多實體、多鏈、多來源。
- ERP API connectors。
- 更完整的報表與審計工具。

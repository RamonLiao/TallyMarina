# 首版資料模型與會計規則引擎設計

可以，首版我建議你把它設計成一個「**事件驅動的數位資產子分類帳**」：先把 Sui / CEX / 託管的原始資料標準化成統一事件，再由會計規則引擎把事件轉成 journal entries，最後輸出到 ERP。這種設計最符合 crypto subledger 的主流架構：資料收集、對帳、成本基礎計算、GL export 四段式流程。[^1][^2][^3]

同時，規則引擎不要直接綁死在單一準則，而應該採「**政策模板 + 規則評估 + 人工覆核**」三層結構。因為數位資產在 IFRS / GAAP 下的分類、衡量和揭露常常依資產性質、用途、持有主體與公司政策而變動，規則必須可配置，不能硬編碼。[^4][^5][^6]

## 首版資料模型

最小可行資料模型建議拆成 10 個核心實體，先不要過度抽象，但一定要保留 audit trail 與 replay 能力。


| 實體 | 作用 | 重要欄位 |
| :-- | :-- | :-- |
| Entity | 法人 / 帳務主體 | entity_id, base_currency, accounting_standard |
| AccountSource | 錢包 / CEX / 託管 / 銀行來源 | source_id, source_type, provider, entity_id |
| RawTransaction | 原始輸入資料 | raw_tx_id, source_id, source_ref, raw_payload, observed_at |
| NormalizedEvent | 標準化後事件 | event_id, event_type, asset_id, quantity, timestamp, counterparty |
| PositionLot | 成本批次 | lot_id, asset_id, acquired_qty, remaining_qty, unit_cost |
| PricePoint | 定價資料 | asset_id, price_ccy, price, timestamp, source |
| PolicySet | 會計政策模板 | policy_set_id, standard, basis_method, valuation_policy |
| MappingRule | 事件到科目的映射規則 | rule_id, event_type, conditions, debit_account, credit_account |
| JournalEntry | 子分類帳分錄 | je_id, entity_id, period, status, source_event_id |
| ReconciliationRecord | 對帳與例外 | recon_id, source_balance, book_balance, variance, status |

這樣的模型對應了典型 subledger 的需求：先把異質來源資料統一，再做分類、估值、lot tracking、分錄輸出與對帳。[^3][^6][^1]

## 核心資料流

建議整體資料流固定成 6 步，這樣系統才可重跑、可審計：

1. 收進 RawTransaction。
2. 轉成 NormalizedEvent。
3. 補上 PricePoint 與 FX。
4. 套用 lot / cost basis。
5. 規則引擎產出 JournalEntry。
6. 對帳後 export 到 GL / ERP。[^7][^2][^1]

重點是 **RawTransaction 永遠不覆蓋**，NormalizedEvent 和 JournalEntry 都要版本化。因為審計時你一定會被問：「這筆數字是從哪個原始交易、哪個規則版本、哪個價格來源算出來的？」[^7][^3]

## 建議的事件模型

首版不要一開始就追求覆蓋所有 DeFi 複雜場景，而是先建立一套可擴充的事件 taxonomy。至少先有下面這些：

- ASSET_RECEIPT：收款、充值、airdrop、外部轉入。
- ASSET_DISPOSAL：付款、提出、外部轉出、出售。
- INTERNAL_TRANSFER：自有錢包 / 託管之間移轉。
- TRADE_BUY：用法幣或其他資產買入。
- TRADE_SELL：出售資產。
- SWAP：鏈上 token swap。
- FEE_GAS：gas fee / network fee。
- STAKING_DEPOSIT：質押投入。
- STAKING_REWARD：staking reward 收入。
- STAKING_WITHDRAWAL：解除質押。
- CUSTODY_MOVE：託管移轉。
- UNCLASSIFIED_CONTRACT_INTERACTION：暫時無法自動理解的合約互動。

這符合市場上對 transaction categorization 的實務需求：先把 transfer、trade、income、expense、DeFi event 轉成可會計化事件。 你的 AI 可以先在最後一類做輔助分類，而不是一開始就讓 AI 決定所有事件。[^8][^1]

## 建議的 normalized event schema

每筆 NormalizedEvent 我建議至少有這些欄位：

- `event_id`
- `entity_id`
- `source_id`
- `raw_tx_id`
- `chain`
- `protocol`
- `event_type`
- `asset_symbol`
- `asset_address`
- `quantity`
- `direction`（in / out / neutral）
- `wallet_address`
- `counterparty_type`（customer / vendor / exchange / self / protocol / unknown）
- `counterparty_ref`
- `tx_hash`
- `block_time`
- `economic_purpose`
- `confidence_score`
- `review_status`
- `tags[]`

其中 `economic_purpose`、`confidence_score`、`review_status` 很重要，因為這三個欄位就是 AI 建議與人工覆核協作的接口。[^8][^3]

## 成本批次與估值模型

數位資產會計不能只記流水，必須做 lot tracking 和 event-time pricing。主流 subledger 也都把 fair market value 與 cost basis lot tracking 當成核心能力。[^9][^1]

所以 `PositionLot` 至少要支援：

- acquisition timestamp
- acquisition source event
- acquired quantity
- remaining quantity
- acquisition unit cost
- functional currency
- cost basis method used

支援的方法先做：

- FIFO
- WAC / weighted average

之後再補：

- LIFO
- HIFO
- Specific ID。[^1][^9]

首版先做 FIFO + WAC 已足夠跑多數早期客戶。因為 MVP 的重點是能穩定出帳，不是一次追求所有 tax optimization 場景。

## 會計規則引擎設計

規則引擎建議採 3 層：

### 第一層：分類規則

目標是把 NormalizedEvent 轉成 accounting treatment candidate。
例如：

- `event_type = INTERNAL_TRANSFER` 且 `source.owner = same_entity`
    - 不認列損益，只移動科目位置。
- `event_type = STAKING_REWARD`
    - 認列收入候選。
- `event_type = SWAP`
    - 拆成 disposal + acquisition + fee 三段會計影響。

這一層對應的是 transaction categorization，自動化程度高，但要保留人工覆核入口。[^1][^8]

### 第二層：政策規則

目標是決定在 IFRS / GAAP / 客戶政策下，怎麼認列與衡量。
例如：

- `asset_class = stablecoin` 且 `policy.stablecoin_treatment = financial_asset`
    - 走金融資產科目。
- `asset_class = token` 且 `policy.crypto_classification = intangible`
    - 走無形資產路線。
- `event_type = STAKING_REWARD`
    - 依政策認列為其他營業收入或投資收益。

這層要明確把「準則」和「公司政策」分開。因為 IFRS / GAAP 給的是框架，但最終落科目通常還要看公司會計政策與 chart of accounts。[^5][^6][^4]

### 第三層：分錄生成規則

這層才真正產出 debit / credit lines。
例如：

- 外部收到 USDC：
    - Dr Digital Assets - Stablecoin
    - Cr Customer Liability / Revenue clearing（依業務性質）
- 支付供應商 USDC：
    - Dr Expense / Prepayment / AP
    - Cr Digital Assets - Stablecoin
- 內部錢包轉帳：
    - Dr Wallet B Holding
    - Cr Wallet A Holding
- Swap：
    - Dr New Asset
    - Dr / Cr Gain or Loss on Disposal
    - Cr Old Asset
    - Dr Fee Expense
    - Cr Asset / Cash for fee

這跟 Oracle / Dynamics 這類傳統 subledger 邏輯一致：來源文件先進 subledger，再由 accounting rules 與 posting profile 生成分錄。[^6][^7]

## 規則資料結構

規則不要寫成一堆 if/else 散在程式裡，建議做成可儲存、可版本化的結構，例如：

```json
{
  "rule_id": "swap_disposal_ifrs_v1",
  "applies_to": {
    "event_type": "SWAP",
    "standard": "IFRS",
    "entity_type": "OPERATING_COMPANY"
  },
  "conditions": [
    {"field": "review_status", "op": "=", "value": "APPROVED"}
  ],
  "actions": [
    {"type": "CREATE_LINE", "side": "DEBIT", "account": "DIGITAL_ASSET_NEW", "amount_source": "fmv_received"},
    {"type": "CREATE_LINE", "side": "CREDIT", "account": "DIGITAL_ASSET_OLD", "amount_source": "carrying_value_disposed"},
    {"type": "CREATE_LINE", "side": "DEBIT_OR_CREDIT", "account": "GAIN_LOSS_DISPOSAL", "amount_source": "balancing_figure"}
  ],
  "version": 1
}
```

這樣做的好處是：

- 可針對不同客戶配置。
- 可針對不同準則切換。
- 規則升級後可重跑歷史期間。
- 能留下完整 rule version audit log。[^4][^8]


## PolicySet 的首版設計

`PolicySet` 我建議至少包含：

- accounting_standard：IFRS / US_GAAP
- functional_currency
- reporting_currency
- cost_basis_method
- stablecoin_treatment
- token_classification_default
- staking_reward_treatment
- impairment_or_fair_value_mode
- fee_expense_policy
- internal_transfer_policy
- chart_of_accounts_mapping_version

這些欄位能決定 80% 的 early-stage 數位資產會計結果。 例如在 IFRS 下，數位資產常涉及 IAS 38 的 cost / revaluation 路徑判斷，而不同公司對 stablecoin 的處理也會有不同政策選擇。[^10][^11][^5][^1]

## JournalEntry 模型

每張 JE 建議拆 header + lines。

### Header

- je_id
- entity_id
- period
- source_event_id
- policy_set_version
- rule_version
- status（draft / approved / posted / reversed）
- memo
- created_at


### Lines

- je_line_id
- je_id
- account_code
- dimension values（entity, cost center, wallet, counterparty, project）
- debit_amount
- credit_amount
- currency
- fx_rate
- reference_asset
- reference_quantity

這樣可以很自然接到 ERP，因為大多數 ERP 的 subledger-to-GL 介面就是 header + lines + dimensions 的模型。[^6][^7]

## 對帳模型

Reconciliation 不能只是報表，要是系統內部的一級資料物件。
建議 `ReconciliationRecord` 至少有：

- recon_scope（wallet / asset / period / source）
- source_balance
- calculated_subledger_balance
- variance_qty
- variance_value
- exception_reason
- assigned_to
- status
- resolved_at

原因是好的 reconciliation 流程要求 accuracy、completeness、timeliness，且要能跨 wallets、custodians、nodes、internal records 比對。[^12][^3]

## 首版 AI 介面怎麼接

AI 在首版不要碰最終分錄，只碰 4 件事：

- 建議 `event_type`。
- 建議 `economic_purpose`。
- 建議 `counterparty_type`。
- 對低信心或異常事件打標。

然後由人工 approve 後，規則引擎才可繼續出 JE。這樣最符合企業會接受的 AI-enabled rules engine 模式，也能避免黑箱會計。[^4][^8]

## 我會建議你首版先支援的規則範圍

先只做這 8 類，夠你拿第一批客戶：

- 穩定幣收款。
- 穩定幣付款。
- 內部錢包移轉。
- CEX 充值 / 提現。
- 現貨買賣 / swap。
- Gas / network fee。
- Staking deposit / reward / withdrawal。
- 期末重估 / fair value adjustment。

這 8 類基本就能涵蓋大部分「正式營運中的 Sui 項目方」與「小型 crypto-active company」的月結工作。[^2][^1]

## 一個實際例子

假設一筆事件是：公司收到 10,000 USDC 作為客戶付款。
系統流程應該是：

1. RawTransaction 讀到錢包收款。
2. NormalizedEvent 分類為 `ASSET_RECEIPT`，asset = USDC，counterparty = customer。[^1]
3. PolicySet 判定 stablecoin 在此公司政策下走 `Digital Asset - Stablecoin` 科目。
4. MappingRule 依 `economic_purpose = customer_receipt` 產生：
    - Dr Digital Assets - USDC
    - Cr Deferred Revenue / AR / Customer Clearing（視情境）
5. ReconciliationRecord 對應 wallet balance 增加 10,000。
6. JE 批次匯出到 NetSuite / Xero。[^2][^7]

同樣的資料流也能處理 staking reward 或 swap，只是規則與 lot tracking 較複雜。

## 首版設計原則

最後給你 5 個原則，這會讓系統比較不像 demo，而像真的能商用的 subledger：

- Raw data immutable，所有後續結果可重跑。[^3]
- Event schema 先穩，再擴充 protocol-specific parser。[^1]
- 規則版本化，所有 JE 都能追溯到 rule version。[^7][^4]
- AI 只做建議，不直接做 final posting。[^8]
- ERP 是 GL of record，你的系統是 subledger，不要一開始就想取代 ERP。[^2][^6]
